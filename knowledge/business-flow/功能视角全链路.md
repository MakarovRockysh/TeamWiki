# 业务流程 · 功能视角全链路

## 概念

任何"功能"在 TeamWiki 中应能一键定位它的**完整上下文**：

```
功能 (Feature)
  ├── 会议 (Meetings)        — 立项 / 评审 / 复盘
  ├── PRD (Requirements)     — 当前版本 + 历史版本
  ├── 原型 (Prototypes)      — 高保真原型 / 流程图
  ├── 代码 (Code)            — PR 列表 / 关键文件
  ├── 测试 (Tests)           — 用例集 / 覆盖率
  ├── 运营 (Ops)             — 上线后的异常 / 复盘
  └── 对话 (Dialogues)       — 关键 AI 对话归档
```

## 数据如何串起来

### ① 文件级 frontmatter（人工标注）

```yaml
---
feature: feature-builder-v1
related:
  - knowledge/prd/项目基础包构建器-PRD.md
  - knowledge/conversation/2026-05-03-Sprint1主壳搭建过程.md
---
```

### ② Agent 自动抽取（Sprint 5+）

- 知识库 Agent 定期扫描 markdown，识别"功能名 / PR / 文件路径"等实体。
- 维护 `data/relations.json`：`{ feature: [...related-files] }`。
- 出现冲突时进入"待人工确认"队列。

## Wiki 中的呈现

- **`/graph/`** — 以功能为中心的辐射图谱（demo 已实现静态版本）。
- **`/feature/<name>/`**（Sprint 4 计划）— 该功能的全链路时间线视图。
- **`/search/`** — 搜功能名时优先返回 `/feature/<name>/` 入口。

## 五阶段闭环（与 Sam 哥讨论一致）

| 阶段 | 沉淀物 | TeamWiki 子目录 |
|------|--------|----------------|
| 项目初期 | 会议纪要（含人员关联） | `knowledge/conversation/` |
| 设计 | 原型 + 流程 + 设计规范 | `knowledge/business-flow/` + `rules/ui-design.md` |
| 编码 | 编码规范 + 上下文存储 | `rules/ai-coding-behavior.md` + `.cursor/rules/` |
| 测试 | 测试用例 + 测试 Agent 输出 | `knowledge/tech/` (测试方案) + Agent 自动归档 |
| 运营 | 异常 / 监控 / 复盘 | `knowledge/conversation/` (踩坑) + `release/` |

## 关键词

> 功能视角, feature view, 五阶段闭环, 强关系索引, frontmatter related, 知识库 agent
