# 知识库写作规范

向 `knowledge/` 新增内容时遵循本规范。目标：**写入即可被搜到、可被引用、可被 Agent 维护**。

## 1. 先确定目录归属

| 子目录 | 进入条件 |
|--------|---------|
| `prd/`           | 是产品需求 / 模块字段 / 验收要点 |
| `plan/`          | 是建设方案 / 路线图 / 多阶段执行计划 |
| `tech/`          | 是技术决策 / 架构 / 数据模型 / 接口 |
| `business-flow/` | 是业务流程 / 全链路视图 |
| `conversation/`  | 是与人 / AI 的对话归档（带日期） |
| `release/`       | 是发布说明（带日期） |
| `others/`        | 都不属于以上 → 进入此处，每月梳理 |

## 2. 文件命名

| 类型 | 格式 |
|------|------|
| `conversation/` `release/` | `YYYY-MM-DD-<主题>.md` |
| `prd/` `plan/` `tech/` `business-flow/` | `<主题>.md`（不带日期，常驻） |

## 3. 内容结构（推荐）

```markdown
---
owner: <git-username>
tags: [...]
related: [...]
updated: YYYY-MM-DD
---

# 标题（与文件名呼应）

## 1. 背景 / 范围
（一句话定位）

## 2. 主体内容
（用表格 / 代码块 / 列表，避免长段落）

## 3. 涉及文件 / 模块
（便于跳转）

## 关键词
> word1, word2, ...
```

## 4. 必填关键词行

每个 `.md` **结尾**写一行：

```
> 关键词: <空格分隔的检索词>
```

便于 grep 与 LLM 识别。

## 5. 链接其他文件

- 用相对路径：`knowledge/conversation/2026-05-03-xxx.md`
- 文档间互引时双向写明（被引用方也写一条 `related`），便于关系图谱

## 6. 不要做的事

- ❌ 写流水账（"今天和某某聊了一下"），写就要写结论
- ❌ 把代码大段贴入正文（用代码块 + 行号引用文件即可）
- ❌ 把多个不相关主题塞到一个文件（拆成多个）
- ❌ 把 `.docx` `.pdf` 直接放进 `knowledge/`（用 `others/` 或 `assets/` 暂存，正文用 markdown 引用）

## 7. 何时归档对话

由用户口头触发：「对话归档」「记一下」「归档当前对话」。详见 `knowledge/conversation/README.md`。

## 8. AI Agent 自动写入（Sprint 5+）

未来知识库 Agent 会自动从飞书 / 微信 / IDE 抽取内容写入：

- 自动写入也必须遵循上述规则
- Agent 写入的文件 frontmatter 中应有 `source: agent-<name>`
- 出现冲突时进入"待人工确认"队列
