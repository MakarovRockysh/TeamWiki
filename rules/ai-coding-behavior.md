# AI 辅助编码行为规范

本规范用于约束 Cursor / Claude Code / Codex 在 TeamWiki 仓库中的编码、评审、重构行为。目标：**减少错误假设、过度设计、无关改动、不可验证的交付。**

## 1. 编码前先澄清

不要替用户做假设。遇到需求歧义、上下文不足、实现路径有多种选择时，先说明假设、列出取舍，必要时确认。

- 不确定时先问，不要猜。
- 存在多种理解时先呈现差异。
- 发现更简单的方案时主动说明。
- 无法判断业务含义时，暂停并指出不清楚的地方。

## 2. 简洁优先

只实现当前需求需要的功能。不额外增加未要求的抽象、配置项、兼容层或过度错误处理。

- 不为一次性逻辑创建复杂抽象。
- 不添加需求之外的功能。
- 不为"未来可能需要"提前设计复杂结构。
- 实现明显可以更短、更直接时，应优先简化。

## 3. 精准修改

只修改与任务直接相关的文件和代码。不顺手重构、格式化、删除不理解的旧代码或改动无关注释。

- 匹配项目现有风格，即使个人偏好不同。
- 不改动与需求无关的相邻代码。
- 不删除预先存在的疑似废弃代码，除非明确要求。
- 若本次改动造成无用导入 / 变量 / 函数，应同步清理。

## 4. 目标驱动执行

每个任务都要有可验证的完成标准。

- 修复问题时，先明确复现条件与预期结果。
- 新增功能时，明确用户可见行为与验收点。
- 重构时，确保重构前后行为一致。
- 多步骤任务给出简短计划，并逐项验证。

## 5. 与本项目其他规范的关系

- UI 改动以 `rules/ui-design.md` 为准
- 知识库写作以 `rules/knowledge-writing.md` 为准
- 规则编写以 `rules/rules-writing.md` 为准
- 输出格式以 `rules/output-format.md` 为准
- 自动生效版本同步维护在 `.cursor/rules/karpathy-guidelines.mdc`

## 6. TeamWiki 特有约束

- **不绑定单一工具**：不要写只能在 Cursor 里跑的代码（除非明确是 `.cursor/rules/`）。
- **markdown 是真源**：所有数据来自 `knowledge/` 下的 `.md`，不要把数据写在数据库 / KV 而绕过 git。
- **静态优先**：能用纯静态实现的不引入构建工具；如必须引入（Sprint 4+），需在 PR 中说明取舍。
- **Layout 自动注入**：新增模块页不要手写 nav / sidebar / footer，引入 `scripts/layout.js` 即可。
