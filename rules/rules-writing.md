# 规则编写规范（meta）

新增 / 修改 `rules/*.md` 或 `.cursor/rules/*.mdc` 时遵循本规范。**这是关于"规则的规则"。**

## 1. 写规则前先问 3 个问题

1. **它解决什么问题？** — 没有具体问题不要写规则。
2. **它会被谁读？** — 是给人读、给 AI 读、还是两者都要？决定写法。
3. **它能被验证吗？** — 不能验证的规则会被忽略。

## 2. 一条规则的最小结构

```markdown
## <规则名>

**适用场景**：<什么时候触发>
**规则**：<明确的 do / don't>
**理由**：<为什么这条规则存在 / 它防止了什么>
**例外**：<什么情况下可以不遵守>
```

最重要的是**理由**。无理由的禁令会让人产生抵触，进而被忽略。

## 3. 给人读的规则 → `rules/*.md`

- 用完整 markdown
- 可以长（必要时分章节）
- 用例子说明"做对" / "做错"
- 文件结尾必填关键词行

## 4. 给 Cursor 读的规则 → `.cursor/rules/*.mdc`

- 必须有 frontmatter，至少包含：

```yaml
---
description: <一句话说明>
globs: ["src/**/*.tsx"]    # 适用文件
alwaysApply: true          # 是否始终生效
---
```

- 正文要**短而精**：Cursor 上下文有限，长规则不会被完全读
- 一条 `.mdc` 文件聚焦一个主题
- 不要把 `rules/*.md` 全文 copy 进来；用引用：`参考 rules/ui-design.md §6`

## 5. 给 Codex / Claude 读的规则

- 在 `AGENTS.md` / `CLAUDE.md` 中分章节呈现
- 由 `/builder/` 或 Sprint 5 的 Agent 自动从源 markdown 转换生成
- **不要**手写 `AGENTS.md` / `CLAUDE.md`，避免与源 markdown 不同步

## 6. 修改既有规则

- 修改前看一眼这条规则被引用的地方（grep）
- 必要时在文件结尾追加"变更记录"小节
- 关键变化（如禁令变松 / 变严）需在 PR 中明确说明

## 7. 规则的退潮

- 每季度评审一次：哪些规则被违反多次但没事 → 这条规则可能没用，删除或弱化
- 哪些规则反复被解释 → 可能不够清晰，重写

## 8. 写规则时的措辞

- ✅ "**禁止** XXX，因为 YYY"
- ✅ "实现 XXX 时**必须** YYY"
- ⚠️ 慎用 "**应该**" "**最好**"（语气模糊，难执行）
- ❌ "**或许**可以 ..."（这种话不应该出现在规则里）

## 关键词

> 规则编写, meta, .cursor/rules, mdc, frontmatter, 规则退潮
